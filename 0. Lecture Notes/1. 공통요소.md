# 커서 핵심 기능과 Rules

## 커서의 핵심 기능
1) `AI 페어 프로그래밍`: 코드를 치면 AI가 자동으로 **추천 코드**를 주고 **Tab키로 선택**하는 형식 

2) `바이브 코딩`: **자연어를 기반**으로 프롬프트를 작성하는 형식. <- 우리가 할 방식

## Rules

### Rule 이란?
- 커서의 기능에 `Rules` 라는 기능이 있음  
- 반복되는 프롬프트를 매번 똑같이 사용하는게 아니라 설정에서 미리 등록하여서 사용
- 모든 프롬프트에 동일한 rules가 첨부 되어서 입력이 적용되도록 하기 위함

### 종류
- `User Rules`: **모든** 프로젝트에 공통적으로 들어가는 조건 
  - ex. 주석은 간결하게 작성해줘. 응답은 한국어로 해줘.
- `Project Rules`: **각각의** 프로젝트에 필요한 특정 조건 
  - ex. .flex-box로만 구현해줘.

### 설정/실행 방법
- 커서 오른쪽 윗 부분 **톱니바퀴** 선택 > **Rules & Memories** 선택
  - `User Rules`: 
    - 1. **Add Rule**
    - 2. 원하는 규칙 입력
    - 3. 모든 프롬프트가 실행될 때 마다 **자동**으로 첨부되어서 프롬프트가 실행됨
  - `Project Rules`: 
    - 1. **Add Rule**
    - 2. **Rule Name** 입력
    - 3. 언제 실행될 지에 맞춰서 원하는 규칙 입력([Rule 파일](1-1.%20Rules%20파일))
    - 4. File Explorer에서 **.curosr/rules** 밑에 rule 잘 생성됐는지 확인
    - 5. 프롬프트 쓰고 **@ 버튼** 눌러서 적용하고 싶은 **rule 선택**하면 됨.

### Rule 파일 설명
- AI가 더 Rule을 잘 인지하게 하는 방법
  - 구분선을 섹션별로 나누기 (=======)
  - 그룹핑 해주기 (1, 2, 3, ...)
  - AI가 잘 이해하는 마법의 단어 활용: `step-by-step`, `구조를 먼저 분석` 등

#### `01-common.mdc` ([파일](1-1.%20Rules%20파일/01-common.mdc))
1. 공통조건
- `명시된 파일 이외에는 절대로 수정하지 말 것`: 내가 구조나 설계를 먼저 만들고, AI가 안을 채우도록 해야 함. 설계 부분을 AI에게 자유도 안넘겨 주고 버그 최소화 하기 위함.
- `명시하지 않은 라이브러리를 설치하지 말 것`: AI가 실행될 때마다 세션이 달라지기 때문에, 자기 멋대로 라이브러리를 설치해서 적용해버리는 것 방지. 예를 들어, 테스트를 위해 playwright을 쓰고 있는데, 멋대로 JEST 방식으로 테스트 해버림.
- `추후 수정이 쉽도록, 독립적인 부품들의 조립 형태로 구현할 것`: 유지보수 차원

2. GIT 조건
- `작업이 모두 끝나면 "Conventional Commits" 방식으로 이름을 만들어서 한국어로 커밋할 것`: 가장 일반적인 방식인 "Conventional Commits"을 사용하여 커밋. 매 프롬프트가 끝날 때마다 commit이 필요. 한번의 작업이 잘못된 작업일 가능성 있음. 그래서 커밋이 되어야 롤백하는 것 가능.

3. 최종 주의사항
- `피그마 링크가 제공되면, 작업을 시작하기 전에 전체적인 피그마 구조를 먼저 분석하고, step-by-step 으로 구현할 것`: 피그마 전체 구조를 보고 작업하는거랑 그냥 작업하는 건 퀄리티 자체에서 차이가 남.
- `package.json을 확인하여, 사용 가능한 라이브러리와 버전을 먼저 step-by-step 으로 분석할 것`: 리액트나 다른 것 버전 확인 안하고 제공 안하는 기능등을 잘못 구현하는 것을 막기 위함.
- `폴더구조, 라우터구조, HTML, CSS 뼈대 layout 구조를 먼저 step-by-step 으로 분석할 것`: 이미 레이아웃에 있는데도 중복해서 만드는 오류를 방지.
- `모든 작업이 끝나면, step-by-step 으로 전체를 검토하여, 빠진 부분을 채우고 디테일 수정할 것`: 실제로 다 했다고 해도 안되어 있는 경우도 있음. 그래서 검토하라고 하고 빠진 부분 업데이트 하기 위해서 넣음.
- `반드시 마지막엔 build 를 실행하여 완료를 확인할 것`: 사람이 할 때도 마찬가지. 제대로 실행되는지 확인하기 위해.

# Figma와 Cursor 연결 및 코드 자동 생성 
- `Cursor Talk To Figma MCP Plugin`를 이용하여 연결 예정

  <details>
  <summary>MCP란?</summary>

  ### MCP (Model Context Protocol)
  - MCP는 **AI** ↔ **다른 소프트웨어**를 이어주는 **공통 언어/규약**
  - 즉, `Cursor AI`와 `Figma 플러그`인 사이 **연결 다리 역할**
  - Figma에서 디자인 요소를 선택하면, MCP가 그 정보를 **Cursor로 전달**
  - 반대로 Cursor에서 생성/편집한 코드나 설정을 **Figma로 보내는 역할**도 함

  </details>

## Cursor ↔ Figma 연결 준비

**1. Bun 설치 (Windows - Powershell/cmd.exe)**
- 설치 ([참고 링크](https://bun.com/docs/installation#windows)):
  ```bash
  powershell -c "irm bun.sh/install.ps1|iex"
  ```

- 설치확인:
  ```bash
  bun -v  # 안뜬다면 powershell 재시작
  ```

**2. Figma 접속**
- 하단 메뉴에서 `플러스버튼` 누르기
- `Cursor Talk To Figma MCP Plugin` 검색하여 실행

**3. MCP 서버 실행**
- Powershell 에서 아래 명령어 실행하여 MCP 서버 실행
  ```bash
  bunx cursor-talk-to-figma-socket
  ```

- 서버가 제대로 켜지면 터미널에 아래와 같은 메시지 뜸:
  ```bash
  WebSocket server running on port 3055
  ```

- Figma로 돌아가서 `Connect` 버튼 누르기

- 초록색으로 `Connected to server in channel: *****` 뜨면 정상

- 채널 이름 복사해 두기

**4. MCP 설정**
- 피그마에 뜨는 아래와 같은 MCP configuration 복사
  ```bash
  {
    "mcpServers": {
      "TalkToFigma": {
        "command": "bunx",
        "args": [
          "cursor-talk-to-figma-mcp@latest"
        ]ㄴㄴ
      }
    }
  }
  ```

- `Cursor IDE` > `Settings` > `Tools & MCP` 선택

- `Installed MCP Servers` 밑에 `Add Custom MCP` 선택

- 복사한 MCP configuration 내용 그대로 붙여넣고 저장

- Cursor 재시작 후 `Installed MCP Servers` 밑에 `TalkToFigma`가 초록색으로 활성화 된 것 확인

## Figma 파일을 이용하여 코드 자동 생성
*여기서는 예시로로 `color.ts`를 자동 생성하기 위해 [`prompt.101.color.txt`](/0.%20Lecture%20Notes/1-3.%20Constants%20프롬프트%20파일/prompt.101.color.txt)을 사용한다고 가정하고 작성함*

**1. 필요한 폴더 및 파일 미리 생성**
- 예시: 
  ```bash
  📂 프로젝트 루트
  ├── 📂 src
  │   ├── 📂 app
  │   └── 📂 commons                                   
  │       ├── 📂 constants   
  │       │   ├── 📂 prompts 
  │       │   │   └── 📄 prompt.101.color.txt # color.ts 자동 생성을 위해 프롬프트 정리한 파일
  │       │   │
  │       │   └── 📄 color.ts # 자동으로 구현될 파일   
  ```

**2. 프롬프트 실행**
- `필요한 부분 수정`: 프롬프트 파일에서 채널명, 노드ID 같은 부분 수정. 채팅창에서 바로 프롬프트 입력하여 명령해도 되지만, 매번 같은걸 쓰려면 귀찮으니깐 미리 준비한 프롬프트 파일(prompt.101.color.txt)에서 필요한 부분만 변경
- `파일 선택된 것 확인`: 실행하고 싶은 텍스트 파일을 포커스해둔 상태에서 채팅창에 입력하려고 보면 이미 *1 Tab* 이런식으로 준비되어 있음 
- `모델 선택 하기`: **Agent** > Auto 끄고, **Claude-4.5-sonnet** 선택
- `프롬프트 실행`: 채팅창에 **구현** 만 입력해도 되나, 나중에 히스토리 찾기 쉽게 하기 위해서 **commons-constatns-color 구현** 이런 식으로 명령
- `재검토`: 규칙을 제공해도 규칙을 다 안지키는 경우가 있어, 마지막에 **룰 재검토 해줘** 하라고 명령 꼭 해야함!
- `변경된 파일 개수 확인`: 내가 원하는 파일만 수정됐는지 확인 필요. 채팅창 가장 밑에 어떤 파일이 변경되었는지 확인

  <details>

  <summary>채널명, 노드ID 확인 법</summary>

  ### 채널명 확인
  - Figma에서 `Cursor Talk To Figma Plugin` 에 보면 중간에 초록색으로 뜬 채널명(ex. w8ozeas1) 복사

  ### 노드 ID 확인
  - Figma에서 자동 구현하고 싶은 페이지(ex. color foundation) 선택 > 마우스 오른쪽 클릭 > Copy/Paste as > Copy link to selection
  - 복사된 URL에서 `node-id=` 이 부분 찾아서 아이디(ex.1-11935) 복사
  - 아이디에서 `-`는 `:`로 변경(ex. 1-11935 -> 1:11935)하여 입력 (노드 ID를 입력해주면 Cursor가 더 빠르게 Figma 읽어옴)
  </details>

# Constants

<details>
<summary>공통: Figma foundation 이란?</summary>

- 화면을 꾸미는 구체적인 컴포넌트보다 더 아래 단계인, 디자인의 기본 빌딩 블록 ([참고: 바이브코딩.fig > 최종본 > 감정일기장 최종본 > Foundation](/0.%20Lecture%20Notes/1-2.%20Figma%20디자인%20파일))
- 이후, `commons/constants`  폴더랑도 1:1로 매칭 됨.  
- Figma foundation에 포함되는 것들:
  - Colour:
    - Primary, Secondary 같은 브랜드 색상
    - Light / Dark 모드 지원 색상
    - Semantic 색상 (Success, Error, Warning 등)
  - Typography:
    - 폰트 패밀리, 크기, 굵기, 줄 간격 등
    - Heading, Body, Caption 같은 텍스트 스타일
  - Spacing & Layout:
    - Margin, Padding, Grid 단위 (예: 4px, 8px 스케일)
  - Elevation / Shadow
    - 컴포넌트 깊이감을 주는 그림자 규칙
  - Border / Radius
    - 버튼, 카드의 모서리 곡률 (예: 4px, 8px, 16px)
  - Icongraphy 
    - 크기, 라인 두께, 사용 규칙
  - Ratio & Constraints
    - Aspect ratio 규칙 ex. 1:1, 16:9 같은 비율
    - 컨테이너 최대/최소 폭 규정
  - Effects
    - Shadow, Blur, Opacity scale
</details>
